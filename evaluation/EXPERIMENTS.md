# Experimental Evaluation: Web-PDE-LLM vs Baseline Methods

This document defines the complete scientific evaluation protocol for comparing
**Web-PDE-LLM** against two SOTA baselines: **LLM-direct** (pure LLM Python
solver) and **CodePDE** (LLM-assisted Python solver with iterative debugging).

Run the full experiment suite with:
```bash
python eval_pipeline.py --all
```

---

## 1. Baselines

| Method | Description | Backend |
|---|---|---|
| **LLM-direct** | Single-shot LLM generates a NumPy/Python FK solver | Python (CPU) |
| **CodePDE** | LLM generates + iteratively debugs a Python solver | Python (CPU/GPU) |
| **Web-PDE-LLM (ours)** | 5-agent pipeline generates a GLSL WebGL shader | Browser GPU |

All methods receive **identical input**: the FK PDE description from
`pde_descriptions.py` with `tau_d = 0.5714`.

---

## 2. Ground Truth

**Source:** `baselines models/fk_data/tau_d_0.5714/`

| File | Content |
|---|---|
| `IC.csv` | 512√ó512 initial conditions (flattened, u/v/w columns) |
| `UVW_array_data.npz` | Full 3D arrays U, V, W of shape `(11, 512, 512)` |
| `sim_data_T*.csv` | Per-timestep snapshots at T = 831.25, 841.25, ‚Ä¶, 931.25 |

The ground truth was generated by a validated reference solver. All methods
are evaluated against these snapshots using the same initial conditions from
`IC.csv`.

**Note on timing:** The snapshots at T > 800 capture the fully developed
spiral wave regime, which is the hardest and most physically meaningful regime
to reproduce. Errors during transient initialization (T < 100) are not reported
as primary results.

## 3. Metrics

### 3.1 Numerical Accuracy (primary)

Computed per state variable (u, v, w) at each of the 11 time snapshots.

| Metric | Formula | Physical meaning |
|---|---|---|
| **Relative L2 Error** | `‚ÄĖx_pred ‚ąí x_gt‚ÄĖ‚āā / ‚ÄĖx_gt‚ÄĖ‚āā` | Overall field accuracy |
| **RMSE** | `‚ąö(mean((x_pred ‚ąí x_gt)¬≤))` | Average pointwise error |
| **Max Absolute Error** | `max|x_pred ‚ąí x_gt|` | Worst-case error |
| **SSIM** | Structural Similarity Index on spatial field | Spatial pattern fidelity |

> **Primary reporting metric:** Relative L2 Error on `u` (transmembrane voltage),
> averaged across the 11 snapshots. This is the same metric used by CodePDE.

### 3.2 Physiological Metrics (secondary)

Extracted from the voltage time series at the domain centre pixel (256, 256):

| Metric | Definition | Relevance |
|---|---|---|
| **Action Potential Duration (APD)** | Time u > V_c = 0.13 in one cycle (ms) | Core cardiac electrophysiology |
| **Conduction Velocity (CV)** | Wavefront propagation speed (cm/s equivalent) | Clinically measurable |
| **Peak Voltage** | max(u) over simulation | Excitability |
| **Resting Voltage** | min(u) in steady state | Repolarization |
| **Spiral Tip Count** | Number of phase singularities detected | Arrhythmia complexity |

### 3.3 Code Generation Metrics (robustness, N=10 trials)

| Metric | Definition |
|---|---|
| **First-gen Success Rate** | % of runs where generated code runs without error on first attempt |
| **Mean Debug Iterations** | Average number of debug‚Üívalidate loops before passing |
| **Validation Pass Rate** | % of final shaders passing all 7 validation checks |
| **Compilation Error Rate** | % of initial outputs with syntax/compilation errors |

### 3.4 Computational Performance

| Metric | Unit | How measured |
|---|---|---|
| **Time-to-generate** | seconds | Wall-clock from prompt to runnable code |
| **Simulation throughput** | time-units/second | T=100 sim wall-clock |
| **FPS** (WebGL only) | frames/second | Browser frame counter |
| **Peak memory** | MB | `tracemalloc` / `psutil` |
| **GPU vs CPU** | ‚ÄĒ | Whether GPU is utilised |

## 4. Experiment Design

### Experiment 1 ‚ÄĒ Numerical Accuracy (FK, tau_d = 0.5714)

**Goal:** Reproduce the primary quantitative result for the paper.

**Protocol:**
1. Load `IC.csv` ‚Üí reshape to (1, 512, 512) for each of u, v, w.
2. Run each method forward from T=0 to T=931.25 using the identical IC.
3. Sample the prediction at T = 831.25, 841.25, ‚Ä¶, 931.25.
4. Load corresponding `sim_data_T*.csv` ground truth.
5. Compute Relative L2, RMSE, MaxAE, SSIM for u, v, w at each snapshot.
6. Report mean ¬Ī std across the 11 snapshots in Table 1.

**Expected output:** `results/exp1_accuracy.csv`, `figures/exp1_error_maps.pdf`

**Paper table format:**

| Method | L2-u ‚Üď | L2-v ‚Üď | L2-w ‚Üď | SSIM-u ‚ÜĎ | RMSE-u ‚Üď |
|---|---|---|---|---|---|
| LLM-direct | | | | | |
| CodePDE | | | | | |
| Web-PDE-LLM | | | | | |

---

### Experiment 2 ‚ÄĒ Computational Performance

**Goal:** Quantify the real-time advantage of WebGL over Python baselines.

**Protocol:**
1. Run each method for exactly T=100 simulation time units starting from `IC.csv`.
2. Record wall-clock time and peak memory with `time` and `tracemalloc`.
3. For WebGL: measure browser FPS via Playwright `performance.now()` counter
   averaged over 60 seconds of live simulation.
4. Repeat 3√ó and report mean.

**Expected output:** `results/exp2_performance.csv`, `figures/exp2_speedup_bar.pdf`

**Paper table format:**

| Method | Sim time (s) ‚Üď | FPS ‚ÜĎ | Peak mem (MB) ‚Üď | GPU? |
|---|---|---|---|---|
| LLM-direct | | ‚ÄĒ | | No |
| CodePDE | | ‚ÄĒ | | Optional |
| Web-PDE-LLM | | ~60 | | Yes |

---

### Experiment 3 ‚ÄĒ Code Generation Robustness (N=10 trials)

**Goal:** Show the 5-agent pipeline is more reliable than single-shot generation.

**Protocol:**
1. Run each code-generation method 10 independent times on the FK description.
2. For each run, record: compilation success (Y/N), number of debug iterations,
   whether the final code produces physically plausible output (u stays in [0,1]).
3. Compute: first-gen success rate, mean iterations, validation pass rate.

**Expected output:** `results/exp3_robustness.csv`, `figures/exp3_success_bars.pdf`

**Paper table format:**

| Method | 1st-gen success ‚ÜĎ | Mean iterations ‚Üď | Validation pass ‚ÜĎ |
|---|---|---|---|
| LLM-direct | | N/A | |
| CodePDE | | | |
| Web-PDE-LLM | | | |

---

### Experiment 4 ‚ÄĒ Generalizability: Aliev-Panfilov 2V

**Goal:** Demonstrate the pipeline is not FK-specific.

**Protocol:**
1. Run the AP model through the Web-PDE-LLM pipeline (already supported).
2. Run a reference Python AP solver (included in `eval/baseline_runner.py`)
   as pseudo-ground truth.
3. Compute Relative L2 and SSIM for u and v vs the reference.
4. Qualitative: show spiral wave snapshots side-by-side.

**Expected output:** `results/exp4_ap_accuracy.csv`, `figures/exp4_ap_spirals.pdf`

---

### Experiment 5 ‚ÄĒ Spatial Pattern Analysis (Ablation)

**Goal:** Show the generated shader reproduces spiral wave morphology, not just
aggregate statistics.

**Protocol:**
1. From Experiment 1 predictions (u field at T=931.25), detect spiral wave tips
   using phase singularity detection (see `metrics.py`).
2. Compute tip count, mean tip trajectory curvature, inter-tip distance.
3. Compare qualitatively via overlaid trajectory plots.

**Expected output:** `figures/exp5_spiral_tips.pdf`

## 6. Reproducibility Checklist

Before submission, verify:

- [ ] All experiments run from a single command: `python eval_pipeline.py --all`
- [ ] Random seeds fixed: `numpy.random.seed(42)` in all Python baselines
- [ ] Hardware logged: CPU model, GPU model, RAM, OS, Python version
- [ ] LLM model versions logged: Ollama model tag + SHA for each agent
- [ ] Ground truth checksums match: `md5sum baselines\ models/fk_data/tau_d_0.5714/*`
- [ ] Results archived: `results/` directory committed or tarred
- [ ] All figures at 300 DPI, vector where possible
